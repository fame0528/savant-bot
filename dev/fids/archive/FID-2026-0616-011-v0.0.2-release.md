## Lessons Learned

1. **Dual-version projects require version-source disambiguation.** When `VERSION` documents protocol/library-conformity and `Cargo.toml` documents the app version, bumping `VERSION` for an app release is a category error. Document the rule in FID/body so future agents don't recreate the bug.

2. **Release-after-fix is the correct order,** not the other way around. The PR/fix at `b5882d0` is on main. v0.0.2 PATCH-bumps Cargo.toml + records the fix in CHANGELOG and tags. The previous session's commit message correctly identified this as needing a v0.0.2 release.

3. **A patch-bump release that touches 0 source files is valid** when the only change is a default value corrected and tagged. The release is documentation-as-artifact, not just code-as-artifact. CHANGELOG accuracy is part of the engineering discipline.

4. **`write_file` is destructive overwrite, not append.** The recovery path was straightforward (git) but the data-loss risk is real. Future agents should READ existing files BEFORE using `write_file` to add content, or use `str_replace` for inserts.

5. **Auto-archive placement of pre-closable FIDs**: Create the file directly in `dev/fids/archive/` rather than `dev/fids/` + move later. The `.gitignore` rule makes `dev/fids/*` non-tracked; `dev/fids/archive/*` is tracked. Avoiding the move step reduces error surface.

6. **The PAT-in-`~/.gitignore`** outstanding item should be tracked separately as a security follow-up, not block releases. Scope discipline matters: shipping releases with a known follow-up is better than holding releases for a tangential cleanup.

7. **Cargo.lock auto-regenerates on `Cargo.toml` version bumps.** Even a single-character version bump causes cargo to rewrite the lock file. Treat `Cargo.lock` as a side-effect of any `Cargo.toml` change and commit it together.
